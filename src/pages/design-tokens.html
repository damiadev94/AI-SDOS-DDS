Overview

Design tokens are the foundational variables that store every visual decision in the system — colors, typography, spacing, sizing, border radius, and motion. They are organized into two tiers: primitive tokens that hold raw values, and semantic tokens that express intent. All tokens are implemented as CSS custom properties in tokens.css and are available globally across every page in the system.

Primitive Tokens

Primitive tokens store raw design values. They are the source of truth for every visual property but must never be used directly in component styles.

Category Examples Count

Semantic Tokens

Semantic tokens map primitive values to meaningful roles. Components must reference only semantic tokens — never raw primitives.

Token Group Prefix Usage
Text --text-* Foreground color for text elements at different emphasis levels.
Background --background-* Surface colors for page, cards, panels, and overlays.
Border --border-* Stroke colors for dividers, card borders, and input outlines.
Font --font-* Font family, weight, size, and line-height for the type scale.
Spacing --spacing-* Margin, padding, and gap values for layout and components.
Radius --radius-* Border radius for cards, badges, nodes, and other elements.

Usage

Primitive tokens hold raw values such as a specific hex color or a pixel measurement. Semantic tokens reference those primitives and give them meaning — for example, --text-primary maps to the appropriate primitive color token for the default text color. This two-tier approach means that if the base color palette changes, only the primitive values need updating; all semantic references remain intact.

Rule: Components must only reference Semantic Tokens. Never use Primitive Tokens directly in component styles.

Notes

All tokens are defined in tokens.css at the :root level, making them available globally. The token file must be loaded second in the CSS import order, immediately after reset.css and before any layer that consumes token values.
