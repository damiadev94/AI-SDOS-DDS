# AI-SDOS Documentation Design System (DDS)

# 03. Component Library

---

# Objetivo

Definir la arquitectura, clasificación y especificación de todos los componentes reutilizables que conforman el Documentation Design System.

Los componentes representan las unidades fundamentales con las que se construyen páginas, diagramas y documentos.

Todo elemento visual del DDS debe derivarse de uno o más componentes definidos en esta biblioteca.

---

# ¿Qué es un Component?

Un Component es una unidad visual reutilizable con una única responsabilidad claramente definida.

Un componente:

- posee una interfaz consistente;
- encapsula una responsabilidad visual;
- reutiliza Design Tokens;
- puede combinarse con otros componentes;
- no depende de un documento específico.

Los componentes constituyen el bloque básico de construcción del DDS.

---

# Principios

Todo componente debe cumplir los siguientes principios:

- Single Responsibility
- Reusable
- Configurable
- Consistent
- Tool Agnostic
- Accessible
- Token Driven
- Composition over Duplication

---

# Arquitectura

La Component Library se organiza en cuatro niveles.

```
Primitive Components
        │
        ▼
Layout Components
        │
        ▼
Diagram Components
        │
        ▼
Composite Components
```

---

# Primitive Components

Representan elementos visuales básicos.

No contienen lógica documental.

Ejemplos:

- Card
- Badge
- Label
- Divider
- Icon
- Button *(si fuera necesario)*
- Arrow
- Connector

---

# Layout Components

Organizan el contenido.

Ejemplos:

- Header
- Footer
- Sidebar
- Section
- Grid
- Container
- Panel
- Column
- Row

---

# Diagram Components

Representan elementos propios de diagramas.

Ejemplos:

- Node
- Relationship
- Edge
- Legend
- Group
- Swimlane
- Timeline
- Decision
- State

---

# Composite Components

Combinan múltiples componentes anteriores.

Ejemplos:

- Roadmap
- Dashboard
- UML Node
- ER Entity
- Architecture Card
- Phase Card
- Engine Card

---

# Component Classification

| Categoría | Responsabilidad |
|------------|----------------|
| Primitive | Elementos básicos de interfaz |
| Layout | Organización espacial |
| Diagram | Elementos específicos de diagramación |
| Composite | Composiciones reutilizables |

# Reglas

Todos los componentes deberán:

- utilizar únicamente Design Tokens;
- poseer una única responsabilidad;
- evitar dependencias circulares;
- ser reutilizables;
- mantener consistencia visual;
- documentarse antes de implementarse;
- permanecer independientes del contenido.

---

# Versionado

Cada componente podrá encontrarse en uno de los siguientes estados.

| Estado | Descripción |
|----------|-------------|
| Draft | En diseño |
| Stable | Aprobado |
| Deprecated | En proceso de reemplazo |
| Removed | Eliminado del sistema |

---

# Evolución

Los componentes podrán:

- ampliarse;
- componerse;
- especializarse;
- deprecarse.

Nunca deberán modificarse rompiendo compatibilidad sin una nueva versión.
